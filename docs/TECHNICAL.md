# Paper Flow 技术说明

[返回中文 README](../README.md) | [English](TECHNICAL_EN.md)

本文说明 Paper Flow 的推荐架构、LLM 职责、兴趣画像、论文对话、缓存与隐私边界。安装和快速使用见[项目 README](../README.md)，构建和发布见[开发指南](DEVELOPMENT.md)。

## 推荐流程

Paper Flow 采用“候选生成 → 本地评分 → LLM 语义筛选 → 多样性重排”的分层结构，并针对本地优先、单用户、小数据和冷启动场景做轻量化适配。

```text
本地 PDF + 显式反馈 + 人工标签
              ↓
      多兴趣原型与语义画像
              ↓
arXiv 分页候选 → 本地召回 → LLM 筛选 → 新鲜度/MMR → 最终批次
```

### 1. arXiv 分页候选生成

- arXiv 大类只限定搜索空间，不作为用户研究兴趣。
- 用户配置的是最终推荐批次；网络请求每页读取 200 条候选。
- 每个本地检索/重排块为 `max(推荐批次 × 3, 60)` 条。
- 通过阈值的论文不足时继续处理下一块，并按 `offset=200/400/...` 请求后续页。
- 只有最终批次已满或回溯窗口确实耗尽时才停止。
- arXiv 返回 HTTP 429 时采用退避和重试，避免紧密重复请求。

### 2. 加权多兴趣召回

标题与 abstract 使用 TF-IDF n-gram 表示。系统保留多个正负兴趣原型，而不是把所有论文平均成一个容易稀释小方向的中心。

主要权重：

| 来源 | 作用 | 权重 |
| --- | --- | ---: |
| 人工偏好标签 | 强正向原型 | `2.5` |
| “感兴趣”论文 | 强正反馈 | `1.8` |
| LLM 学习的偏好方向 | 正向原型 | `1.8` |
| “还行”论文 | 弱正反馈 | `0.45` |
| “不感兴趣”论文 | 负向原型 | 强负权重 |
| 人工回避标签 | 强负向原型 | 强负权重 |

人工标签权重最高，用户可以直接新增或删除任意文本，不经过自然语言指令解析。

### 3. LLM 语义筛选

LLM 只接收本地召回后的小候选池及其 arXiv metadata，同时读取缓存的兴趣画像、最近的真实正负反馈和人工标签。它返回兴趣概率和明确的拒绝判断。

候选需要满足兴趣概率至少 `0.58` 且没有命中回避方向。系统优先保证相关性；当回溯窗口内确实没有足够合格论文时，最终批次可以少于设置值。LLM 超时、失败或返回结构异常时会回退到本地排序。

### 4. 新鲜度与多样性

新论文获得小幅时间加权。最终选择使用 Maximal Marginal Relevance（MMR）思想平衡相关性与论文间差异，减少同一小方向的近重复论文占满整批。

### 5. 显式反馈闭环

“感兴趣 / 还行 / 不感兴趣”会立即改变下一批的兴趣权重。只有明确选择后，论文才从未来候选中排除；未评价论文可以再次出现。

反馈和行为严格分离：打开 PDF、打开聊天窗口、发送问题或切换聊天历史都不会自动改变兴趣选择。

## LLM 细粒度兴趣与主题词表

### 兴趣画像

LLM 综合以下信息生成一句画像总结和完整、可解释的兴趣方向：

- 最多 12 篇本地 PDF 的截断开头文本；
- 用户对推荐论文的正负反馈；
- 用户手动维护的偏好与回避标签。

结果会过滤 `et al`、`omitted picture`、`learning` 等解析、排版或过于泛化的噪声。兴趣方向数量没有固定为三个的上限。

画像采用事件驱动的后台更新。程序启动、本地论文缓存变化、反馈变化、模型设置变化或人工标签变化时会触发可合并的后台任务；统计页只读取 SQLite 缓存，不在打开页面时等待 LLM。短时间连续变化会被合并，减少重复调用。

### 细粒度主题与语义合并

每篇推荐论文会获得 1–3 个具体主题，例如 `self-supervised vision foundation models` 或 `test-time adaptation for vision foundation models`，而不是 `cs.CV`、`cs.LG` 等大类。

系统维护增量式规范词表：

1. 将现有高频词表提供给 LLM，要求语义等价时复用已有标签。
2. 只为真正不同的研究问题创建新标签。
3. 对输出执行本地规范化，合并大小写、标点、连字符、单复数和词序差异。
4. 将主题生成并入 metadata TL;DR 请求，不额外上传全文。

## 四类模型配置

所有调用使用用户配置的兼容 OpenAI Chat Completions 接口。模型名必须由当前 Base URL 支持。

| 配置 | 实际职责 | 边界 |
| --- | --- | --- |
| 语义筛选模型 | 候选兴趣概率和拒绝决策 | 与文案生成独立，可选择擅长相关性判断的模型 |
| 推荐总结模型 | metadata TL;DR、推荐理由、细粒度主题、已有推荐文案翻译 | 共享同一套结构化推荐内容链路 |
| 兴趣画像模型 | 后台总结本地论文、反馈和人工标签 | 独立缓存、独立刷新 |
| 论文聊天模型 | 基于全文的多轮问答 | 上下文最长，可单独使用更强模型 |

旧版本只有一个模型字段时，四个槽位会先继承旧值，直到用户保存新的细分设置。

## Chat with Paper

论文聊天运行在独立原生窗口中，可拖动、缩放、最小化和恢复。左侧按最近活动显示持久化会话，右侧显示当前论文和消息。

- 第一次发送问题时才会下载并解析论文全文。
- 默认问题是“为我详细介绍这篇论文的方法”。
- system prompt 保持通用，可按用户问题解释方法、实验、公式、假设、对比或局限。
- 最近 16 条消息用于构建多轮上下文。
- 切换历史只读取本地数据库，不调用 LLM。
- Chat with Paper 取代旧的固定“详细 TL;DR”行为。

## 历史与本地检索

“历史与检索”保存所有推送过的论文并支持按时间浏览。搜索在本机索引标题、abstract、metadata TL;DR、旧版详细 TL;DR 和细粒度主题，既接受关键词也接受描述性自然语言。论文聊天历史单独显示在聊天窗口左侧。

两类历史均不会发送给外部搜索服务。

## 数据、缓存与隐私

桌面数据库：

```text
%LOCALAPPDATA%\Paper Flow\state.db
```

- API Key、设置、反馈、推荐记录、画像和聊天历史保存在本机 SQLite。
- 初次推荐和语义筛选只向 LLM 发送 arXiv metadata。
- 为建立兴趣画像，最多发送 12 篇本地 PDF 的截断开头文本；结果按论文库与反馈状态缓存。
- 只有用户主动发送论文聊天问题时，才会处理并发送该论文的截断全文。
- 切换语言只翻译缓存的摘要和推荐原因，不重新发送论文原文。
- 本地 PDF 解析按文件大小、修改时间等状态增量缓存。
- 清除推荐记录与清除本地解析缓存相互独立，均不会删除磁盘上的 PDF。
- 卸载应用不会自动删除用户数据库。

首次桌面启动时，如果发现旧开发版的 `~/.arxiv-daily/state.db`，仅在目标数据库不存在时复制到新目录，不覆盖已有数据。

## 容错与进度

- LLM JSON 输出会经过清理、容错解析和结构校验，降低非严格 JSON 导致的失败。
- 推荐总结失败时保留并显示原始 abstract。
- arXiv 429 使用退避重试；候选不足时区分网络检索页和最终推荐批次。
- 前端显示 arXiv 请求、兴趣初筛、LLM 筛选、总结、全文下载、文本提取和保存等真实阶段。
- 已生成的 TL;DR、推荐理由、翻译、主题和兴趣画像会缓存，避免重复 Token 消耗。
