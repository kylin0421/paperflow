# Paper Flow 技术架构

## 系统边界

Paper Flow 是单用户、本地优先系统。SQLite 是唯一事实来源，arXiv 是论文来源，所有 LLM 调用都使用用户配置的服务和分功能模型。

```text
本地 PDF + 显式评分 + 人工标签
                 │
                 ▼
          细粒度兴趣画像
                 │
arXiv 缓存 ─► 混合召回 ─► LLM 筛选 ─► 配额/MMR ─► 推荐流
                                                       │
PDF 行为 ─► PyMuPDF 或 MinerU ─► 结构化章节 ─► 证据化对话
```

评分与行为完全分离：评分训练推荐；打开 PDF、发送论文问题不会隐式修改偏好。

## 推荐算法 v2

### 候选服务

后台候选服务按 arXiv 分类查询持久化缓存最新 metadata。生成推荐时先消费缓存；最终批次不足才继续网络翻页。

- `batch_size`：最终展示数量；
- `retrieval_batch_size`：一次进入本地召回和 LLM 筛选的候选数量；
- `max_candidates`：确认当前回溯范围无更多合格论文前的最大扫描量。

arXiv 网络分页独立配置。HTTP 错误会持久化失败次数和下次重试时间并指数退避，因此重启或连续点击不会造成 429 紧密重试；后台任务状态也会持久化。

### 兴趣证据与时间衰减

系统保留多个正负兴趣原型，不把所有兴趣平均成一个向量：

| 证据 | 基础权重 | 行为 |
| --- | ---: | --- |
| 人工偏好/回避方向 | 2.5 | 最高优先级，不衰减 |
| LLM 正向兴趣 | 1.8 | 缓存语义画像 |
| LLM 负向兴趣 | 2.1 | 强回避证据 |
| 感兴趣论文 | 1.8 | 按时间衰减 |
| 还行论文 | 0.45 | 弱证据，按时间衰减 |
| 不感兴趣论文 | 1.8 负向 | 按时间衰减 |
| 本地 PDF 开头 | 0.45–0.55 | 冷启动种子 |

行为反馈按可配置半衰期衰减并保留最低权重，使近期兴趣变化生效但不抹除长期历史。人工方向永不衰减。

### 混合召回

第一路使用标题和摘要的词级 TF-IDF unigram/bigram，取最相关正向原型并扣除强负向匹配。第二路使用字符 n-gram TF-IDF，对连字符、词形、缩写和技术短语换序更稳健。用户也可配置 embedding 模型替代字符通道；失败时自动回退本地算法。

两个通道归一化后融合，发布时间只用于打破相关性相近的排序。

### LLM 筛选与自动校准

LLM 只接收本地召回后的 metadata 小池、正负兴趣画像、高权重人工方向和近期反馈示例。它为每篇返回兴趣概率、明确拒绝标记、具体理由和命中的规范兴趣。仅有 “vision”“learning” 等泛化重叠不足以入选。

积累至少 8 个已评分推荐后，系统会根据真实反馈扫描候选阈值，优化精度、召回与负反馈惩罚；此前使用配置默认值。

### 精准、均衡与探索

- **精准**：提高语义门槛，几乎不保留探索位；
- **均衡**：使用校准阈值，保留少量相邻兴趣；
- **探索**：降低相邻方向门槛，提高探索比例，并收紧单主题上限。

最终选择先限制一个兴趣方向的占比，再预留模式对应的探索位，最后使用 MMR 相似度惩罚减少同质论文。这是成熟推荐系统常见的“召回 → 排序 → 后排序约束”结构在本地冷启动论文流中的适配。

## 细粒度兴趣分类

兴趣画像模型生成 `self-supervised vision foundation models`、`test-time adaptation for vision foundation models` 这类具体方向，而不是 `cs.CV`。

生成时提供现有规范标签；本地继续按大小写、标点、单复数、token 重叠和编辑相似度合并。PDF、反馈、人工标签或相关模型设置变化后，后台合并刷新画像。标签数量没有 3 个上限。

## 推荐诊断与离线评估

每轮保存安全设置快照、各路召回/LLM/最终分数、arXiv offset、拒绝理由、命中兴趣、探索标记和最终位置；不会保存 API Key 或私有 Base URL。

后续真实评分会回连到决策记录，计算兴趣命中率、不感兴趣率、空批次率、平均扫描数量和主题覆盖。统计界面可以检查最近每轮的决策路径，使算法升级有可比较指标。

## 长 PDF 论文对话

论文对话使用通用研究助理提示词，默认问题是详细解释方法，也允许追问公式、实验、局限和任意细节。PDF 解析按 SHA-256 指纹缓存：

1. `auto`：本机托管 MinerU 已安装时优先使用，也可使用远程服务；失败回退 PyMuPDF；
2. `mineru`：必须使用所选的本机/远程 MinerU，错误直接显示；
3. `pymupdf`：始终使用本地轻量解析。

托管运行时与 Python 3.13 主应用隔离：随包提供的 `uv` 按需创建应用专属 Python 3.12 环境并安装固定版本 CPU pipeline。本机 Worker 懒启动、跨论文复用、崩溃后重启，并在取消或退出时关闭。因此基础 localhost 部署仍然轻量，而 Windows 用户不需要手动配置 API。

短论文保留完整结构化 Markdown。长论文先生成文档章节地图，再选择完整章节、相邻上下文，以及方法/结果/局限等锚点；所选章节获得 `[S4]` 形式证据编号，回答必须引用，界面显示章节名称。它避免了全文硬截断和细碎小块 RAG 的共同问题。详见 [MinerU 与长 PDF 对话](MINERU.md)。

## 模型职责

| 设置 | 职责 |
| --- | --- |
| 语义筛选 | 候选相关性、拒绝、命中兴趣 |
| 推荐总结 | metadata TL;DR、理由、主题标签、翻译 |
| 兴趣画像 | 后台细粒度兴趣合成 |
| 论文对话 | 结构化全文多轮问答 |
| Embedding（可选） | 混合语义召回 |

旧安装在单独保存各模型前继承共享模型。请求支持超时与重试；JSON 语法失败会进行一次只修复结构的重试，再安全回退。

## 存储、安全与恢复

Schema v3 包含设置、加密 secret、论文、批次、诊断、候选缓存、源状态、本地/解析文档、活动、对话和后台任务。

- API Key 不进入 settings 表；Windows 使用用户绑定 DPAPI，其他系统使用安装实例专属 Fernet 密钥。
- 旧明文 Key 会一次性迁移、加密并执行数据库清理。
- SQLite 使用 WAL、busy timeout、串行写入、启动中断恢复、完整性校验备份和受验证的恢复。
- 仅当用户主动发送论文问题后才处理和发送全文；推荐阶段只使用 arXiv metadata。
- 语言切换只翻译已有缓存文案，不重传论文来源。
- 长任务提供真实阶段进度，并可在网络/模型步骤之间取消。
